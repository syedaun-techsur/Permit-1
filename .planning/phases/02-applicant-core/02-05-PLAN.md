Build the visual lifecycle timeline stepper. This is the primary STAT-02 deliverable.

**Props interface**:
```typescript
interface PermitStatusTimelineProps {
  stages: LifecycleStage[];         // Completed stage entries from API (immutable history)
  currentStatus: ApplicationStatus; // Current application status
  infoRequestNote?: string;         // Shown prominently if additional_info_needed
}
```

**Stage definition and order**:
The timeline always shows these 7 stages in fixed order:
```typescript
const STAGE_ORDER: ApplicationStatus[] = [
  'draft',
  'submitted',
  'under_review',
  'additional_info_needed',
  'approved',     // terminal branch A
  'rejected',     // terminal branch B
];
```

**Stage labels**:
```typescript
const STAGE_LABELS: Record<ApplicationStatus, string> = {
  draft: 'Draft',
  submitted: 'Submitted',
  under_review: 'Under Review',
  additional_info_needed: 'Additional Info Needed',
  approved: 'Approved',
  rejected: 'Rejected',
};
```

**Stage state derivation**:
```typescript
// For each stage:
// - 'completed': exists in stages[] array (i.e., application passed through it)
// - 'current': matches currentStatus AND not in terminal done state
// - 'future': neither completed nor current
// - Special: if currentStatus === 'approved', then 'rejected' is 'muted-terminal'
// - Special: if currentStatus === 'rejected', then 'approved' is 'muted-terminal'
```

**Layout — vertical stepper**:
```
○ Draft                          ← filled circle + green check (completed)
│  entered_at: relative (absolute on hover tooltip)
○ Submitted                      ← filled circle + green check (completed)
│  entered_at: ...
○ Under Review                   ← primary ring (current) + "In Progress" pill
│
○ Additional Info Needed         ← muted gray (future)
│
├─○ Approved                     ← muted gray (future)
└─○ Rejected                     ← muted gray (future)
```

When `approved` or `rejected` reached:
```
○ Approved ✓                     ← filled green (reached terminal)
│
└─○ Rejected                     ← muted gray with strikethrough label (unreached)
```

**Styling**:
- Vertical connector line: `border-l-2` in gray (future), primary color (completed)
- Completed indicator: `w-6 h-6 rounded-full bg-primary flex items-center justify-center` + CheckIcon (Lucide)
- Current indicator: `w-6 h-6 rounded-full border-2 border-primary bg-white` + animated pulse ring
- Future indicator: `w-6 h-6 rounded-full border-2 border-gray-300 bg-white`
- Timestamp: `text-sm text-gray-500` with `title` attribute for absolute date
- "In Progress" label: small badge `bg-primary/10 text-primary text-xs px-2 py-0.5 rounded-full`
- Terminal branching: last two stages rendered with `ml-4` indent and a `└─` or `├─` connector

**Info request note**: If `infoRequestNote` prop is provided AND `currentStatus === 'additional_info_needed'`:
```tsx
<div className="mt-2 p-3 bg-orange-50 border border-orange-300 rounded-lg text-sm text-orange-800">
  <p className="font-semibold mb-1">Additional Information Requested:</p>
  <p>{infoRequestNote}</p>
</div>
```
Render this note immediately after the Additional Info Needed stage row.

**Responsive**: Timeline must be fully readable at 375px width — use text-sm on mobile, no horizontal overflow.

**Timestamps**: use `date-fns` `formatDistanceToNow` for relative display. Add `title={new Date(entered_at).toLocaleString()}` for absolute hover tooltip.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | GET /permits/:id — route param :id from browser URL crossing into NestJS handler |
| db→render | Permit fields (including info_request_note, decision_reason) rendered to DOM — must be owner-scoped |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-24 | Elevation of privilege | GET /permits/:id returns another applicant's permit if applicant_id check bypassed | transfer | Server-side: PermitsService.getById() checks applicant_id === req.user.id (02-01); frontend consumes only what API returns — cannot bypass ownership check |
| T-02-25 | Information disclosure | info_request_note or decision_reason leaked to wrong applicant via URL enumeration | transfer | Same as T-02-24: ownership enforced in PermitsService.getById(); a UUID ID space (v4) is sufficiently large to prevent enumeration — 2^122 possibilities; 403 returned for non-owner |
| T-02-26 | Denial of service | 15-second polling creates excessive API load per active browser tab | accept | 15s interval is a reasonable tradeoff for near-realtime status updates; at scale, WebSocket SSE (Phase 3) will replace polling; accepted for Phase 2 — one request per 15s per active user is within NestJS/Postgres capacity for expected v1 user counts |
| T-02-27 | Tampering | XSS via rendered info_request_note or decision_reason (admin-controlled content in DOM) | mitigate | React DOM rendering escapes HTML by default; info_request_note and decision_reason are rendered inside React text nodes (`{permit.info_request_note}`), never via `dangerouslySetInnerHTML`; enforced in PermitDetailPage.tsx |
</threat_model>
