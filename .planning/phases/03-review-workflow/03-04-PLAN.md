**Step 1 — Extend `frontend/src/api/permits.api.ts`** (already exists from Phase 2):

Add the following methods to the existing permitsApi object:

```typescript
// Reviewer queue
getReviewQueue: (params: { status?: string; permitType?: string; assignment?: string; page?: number; limit?: number }) =>
  axiosInstance.get<{ data: ReviewQueueItem[]; meta: { total: number; page: number; limit: number; totalPages: number } }>('/permits/review-queue', { params }),

// Lifecycle action endpoints
beginReview: (id: string) =>
  axiosInstance.post<ApplicationObject>(`/permits/${id}/actions/begin-review`),

requestInfo: (id: string, infoRequestNote: string) =>
  axiosInstance.post<ApplicationObject>(`/permits/${id}/actions/request-info`, { infoRequestNote }),

respondToInfo: (id: string, responseNote?: string) =>
  axiosInstance.post<ApplicationObject>(`/permits/${id}/actions/respond-to-info`, { responseNote }),

decide: (id: string, outcome: 'approved' | 'rejected', decisionReason: string) =>
  axiosInstance.post<ApplicationObject>(`/permits/${id}/actions/decide`, { outcome, decisionReason }),

// Document archive
getDocumentArchive: (id: string) =>
  axiosInstance.get<{ downloadUrl: string; expiresAt: string }>(`/permits/${id}/documents/archive`),
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→API | Reviewer action forms (infoRequestNote, decisionReason, responseNote) sent to lifecycle action endpoints |
| client→API | Route params (:id in /review/:id) used in API calls — IDOR risk if backend doesn't enforce ownership |
| db→render | Applicant-supplied info response note and reviewer info request note rendered in PermitDetailPage / ReviewDetailPage |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-19 | Elevation of privilege | /review/queue route — applicant navigating to reviewer-only page | mitigate | React Router RoleGuard wrapping /review/* routes redirects non-reviewer/admin users to /unauthorized; backend /permits/review-queue also returns 403 for non-reviewer callers (dual enforcement) |
| T-03-20 | Elevation of privilege | Reviewer calling respondToInfo (applicant-only action) via direct API | transfer | Backend RolesGuard on respond-to-info endpoint enforces APPLICANT role only; frontend simply does not render the respond UI for reviewer role — backend is the authoritative enforcement point |
| T-03-21 | Tampering (XSS) | ReviewDetailPage rendering permit.infoRequestNote / infoResponseNote from DB | mitigate | Both fields rendered via React JSX interpolation `{permit.infoRequestNote}` (never dangerouslySetInnerHTML); React escapes all string content |
| T-03-22 | Information disclosure | ReviewDetailPage showing applicantEmail/applicantPhone to wrong role | transfer | Backend GET /permits/:id conditionally returns reviewer-visible fields only when caller role is REVIEWER or ADMIN (implemented in PermitsService.getById() in Phase 2 + Phase 3 extension); frontend rendering these fields is presentation-only |
| T-03-23 | Denial of service | Oversized decisionReason or infoRequestNote in frontend forms | mitigate | React Hook Form Zod schema validates MaxLength(2000) client-side on DecisionModal and RequestInfoModal before API call; backend class-validator @MaxLength(2000) enforces independently on DTOs |
</threat_model>
